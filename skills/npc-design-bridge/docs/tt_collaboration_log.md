# TT 协同记录 / TT Collaboration Log

本文档记录 NPC Design Bridge Skill 与 TT 设计学院专家分身的协同方案。

---

## 协同架构

```
NPC Design Bridge (主 Skill)
    │
    ├── 阶段一：设计桥接
    │   ├── TT 服务设计专家 → 体验目标审查
    │   └── TT 视觉叙事专家 → 角色概念卡审查
    │
    ├── 阶段二：玩法锚定
    │   ├── TT 游戏策划专家 → 核心循环 + 功能定位审查
    │   └── TT 叙事设计专家 → 对话样例审查
    │
    └── 阶段三：作品集成
        └── TT 设计评审/作品集专家 → 最终 Bible 审查
```

---

## 专家协同矩阵

| 阶段 | TT 专家分身 | 介入时机 | 判断标准 | 产出增量 |
|-----|-----------|---------|---------|---------|
| 设计桥接 | 服务设计专家 | 用户只有模糊情绪、角色印象时 | 是否能转为玩家体验目标和接触点 | 体验目标句、接触点地图 |
| 设计桥接 | 视觉叙事专家 | 角色概念卡阶段 | 外观、语气、隐喻是否统一，是否有辨识度 | 角色识别度建议 |
| 玩法锚定 | 游戏策划专家 | 核心循环 + 功能定位阶段 | NPC 是否服务玩家行为，功能在核心循环中是否成立 | 核心循环审查、功能定位建议 |
| 玩法锚定 | 叙事设计专家 | 对话样例阶段 | 对话是否承担玩法功能，是否符合 NPC 语气一致性 | 功能型对话改写 |
| 作品集成 | 设计评审/作品集专家 | 最终整合 + 自检阶段 | 文档是否可展示、可继续开发、达到作品集标准 | Bible 修改建议、展示建议 |

## 协同原则

1. **增量而非替代**: TT 专家的作用是"增强"主 Skill 的判断力，不是替代主 Skill 的引导流程
2. **按需调用**: 不是每个用户每步都需要专家参与。只有在以下情况触发：用户产出明显偏弱 / Skill 不确定是否通过检查点
3. **保持角色一致性**: TT 专家的输出需要与主 Skill 的"助教"语气保持一致，不能出现两个不同性格的 AI 同时对话
4. **保留决策权给用户**: TT 专家的建议是"专业意见"，用户可以接受或拒绝
